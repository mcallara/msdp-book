# 4 AI-Assisted Software Development

The last three chapters introduced you to version control, Python packaging,
testing and debugging. This chapter is about something that, on the surface,
seems to make all of that obsolete: **AI coding agents** that can write tests,
implement features, and even open pull requests on your behalf.

The thesis of this chapter is the opposite. The arrival of capable agents
makes the practices we have seen so far *more* valuable, not less. The reason is
that an agent that writes plausible-looking code at high speed needs an
environment that can quickly tell *correct* from *plausible*. That environment
is exactly what version control, tests, and a debugger provide.

Although in the course we will use use **GitHub Copilot in VS Code**, the
ideas transfer to other tools, like Codex, Claude Code, Cursor, Windsurf.


```{admonition} Vibes vs. engineering
:class: tip
Andrej Karpathy popularised the term *vibe coding* in 2024 to describe a
loose, conversational style of working with an AI model. It works for
prototypes and breaks down for anything you have to maintain. A December
2025 field study by UC San Diego and Cornell of professional developers
working with coding agents concluded: **"professional developers
don't vibe, they control."** Most of this chapter is about *what control
looks like* when your collaborator is an LLM.
```

## Three eras of working with LLMs

In the three years between 2023 and 2026 the way practitioners describe their
work with language models has shifted twice.

```{mermaid}
flowchart LR
    A["2023<br/><b>Prompt engineering</b><br/>get the wording right"]
    B["2025<br/><b>Context engineering</b><br/>get the right things into the window"]
    C["2026<br/><b>Harness engineering</b><br/>build the environment around the model"]
    A --> B --> C
```

Each shift was a response to the same observation: smarter models do not by
themselves produce reliable systems. Reliability comes from the environment
around the model, this is, the specs, tests, memory, and tools that shape its work.

## Some common failure modes

In this section we will discuss some of the most frequently encountered failure modes in AI coding assistants. If you have used AI coding, you have probably seen some of these failure modes yourself. 


| # | Failure mode | What it looks like |
|---|---|---|
| 1 | **Misalignment / goal drift** | Agent solves the wrong problem; intent under-specified or silently mutated mid-task. |
| 2 | **Hallucination** | Invented APIs, fabricated parameters, fictional library names. Now rare in raw form; survives as invented *capabilities* — flags or kwargs that do not exist on the installed version. |
| 3 | **Context loss & drift** | Agent forgets prior decisions; loses track of state across sessions or compactions. ~65% of enterprise agent failures attributed here. |
| 4 | **Token / context bloat** | Verbose outputs and unbounded context accumulation; larger windows hide rather than solve the problem. |
| 5 | **Verbosity & over-engineering** | Agent adds unrequested abstractions; "one-shots" entire apps; generates more code than was asked for. |
| 6 | **Codebase entropy** | Architectural drift, duplicated logic, broken patterns accumulating faster than humans can review. |
| 7 | **Decision amnesia** | Architectural rationale evaporates when sessions end; future agents re-litigate settled questions. |
| 8 | **Destructive / irreversible actions** | Over-broad permissions → dropped databases, mass deletes, leaked secrets. (Canonical: the Replit DB-deletion incident, July 2025.) |
| 9 | **Prompt injection & memory poisoning** | Hostile instructions embedded in docs, tools, or RAG indexes; persistent corruption of agent memory.|

An example from the real world:

The **Replit incident** (failure mode 8) is a frequently cited example. In
July 2025 an autonomous coding agent, given broad shell access during a
multi-day build, decided that the production PostgreSQL database "looked
unused" and dropped it. There was no malice and no exotic prompt — just an
agent acting inside a permission boundary that should never have been that
wide.

## How to deal with the failure modes?

These failure modes are not inherent to the models themselves. They are artifacts of the environment in which the model operates. With the right practices and tools, you can mitigate or even eliminate many of these failure modes. The table below summarizes some of the key practices and artifacts that can help you deal with each failure mode.


| AI fails when… | Practice | Artifact |
|---|---|---|
| …it solves the wrong problem (1) | **Spec-driven thinking** | a written spec; Spec-Kit; `/grill-me`-style interview |
| …it produces plausible but wrong code (2) | **Test-Driven Development** | a test that pins behaviour, written before the implementation |
| …it forgets conventions across sessions (3) | **Project memory** | `AGENTS.md`, `CLAUDE.md`, `copilot-instructions.md` |
| …it bloats the context window (4) | **Parsimony** | compressed memory; caveman-style prompts |
| …it over-engineers (5) | **YAGNI; vertical slices** | small specs; per-task scoping |
| …it accelerates the codebase into mud (6) | **Design discipline** | ADRs; "zoom out" sub-agents; periodic refactors |
| …it forgets *why* (7) | **Decision logs** | ADRs; persistent memory tools |
| …it runs destructive commands (8) | **Least privilege** | tool allowlists; sandboxes; git guardrails |
| …it can't see the right context | **Tool use & retrieval** | MCP servers; custom tools |


## The tool landscape

This chapter uses **GitHub Copilot in VS Code** for every hands-on step. Copilot is one of several roughly equivalent options as of 2026:

- **GitHub Copilot** (Microsoft) — what we use here. Strong VS Code integration; agent mode shipped in 2025.
- **OpenAI Codex** — cloud-hosted coding agent; same `AGENTS.md` standard.
- **Anthropic Claude Code** — terminal-native agent; introduced the `SKILL.md` and `CLAUDE.md` conventions we will see in §4.2.
- **Cursor**, **Windsurf** — VS Code forks with their own agent UIs.

The patterns we will learn — specs, tests, project memory files, tool allowlists — work across all of them. The names of files (`AGENTS.md` vs `CLAUDE.md`), the keyboard shortcuts, and the chat panel layout differ; the underlying engineering does not.
