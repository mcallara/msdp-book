# 4 AI-Assisted Software Development

The last three chapters introduced you to version control, Python packaging,
testing and debugging. This chapter is about something that, on the surface,
seems to make all of that obsolete: **AI coding agents** that can write tests,
implement features, and even open pull requests on your behalf.

The thesis of this chapter is the opposite. The arrival of capable agents
makes the practices from Chapters 1–3 *more* valuable, not less. The reason is
simple: an agent that writes plausible-looking code at high speed needs an
environment that can quickly tell *correct* from *plausible*. That environment
is exactly what version control, tests, and a debugger provide.

We will use **GitHub Copilot in VS Code** for every demo in this chapter. The
ideas transfer to other tools — Codex, Claude Code, Cursor, Windsurf — but
having a single, concrete tool keeps us out of catalogue-shopping mode.


```{admonition} Vibes vs. engineering
:class: tip
Andrej Karpathy popularised the term *vibe coding* in 2024 to describe a
loose, conversational style of working with an AI model. It works for
prototypes and breaks down for anything you have to maintain. A December
2025 field study by UC San Diego and Cornell of professional developers
working with coding agents concluded: **"professional developers
don't vibe, they control."** Most of this chapter is about *what control
looks like* when your collaborator is an LLM.
```

## Three eras of working with LLMs

In the three years between 2023 and 2026 the way practitioners describe their
work with language models has shifted twice.

```{mermaid}
flowchart LR
    A["2023<br/><b>Prompt engineering</b><br/>get the wording right"]
    B["2025<br/><b>Context engineering</b><br/>get the right things into the window"]
    C["2026<br/><b>Harness engineering</b><br/>build the environment around the model"]
    A --> B --> C
```

Each shift was a response to the same observation: smarter models do not by
themselves produce reliable systems. Reliability comes from the environment
around the model — the specs, tests, memory, and tools that shape its work.

## A taxonomy of agent failure modes

Practitioners between 2024 and 2026 converged on the nine recurring failure
modes below. You are not expected to memorise all nine. You are expected to
recognise them when you meet them in the wild.

```{admonition} The pedagogical contract
:class: important
This is the **map**. The next two sections will live in **four** of these
regions and address them in depth. The other five are signposts: when you meet
them at work, you will recognise the shape of the problem.
```

| # | Failure mode | What it looks like |
|---|---|---|
| 1 | **Misalignment / goal drift** | Agent solves the wrong problem; intent under-specified or silently mutated mid-task. |
| 2 | **Hallucination** | Invented APIs, fabricated parameters, fictional library names. Now rare in raw form; survives as invented *capabilities* — flags or kwargs that do not exist on the installed version. |
| 3 | **Context loss & drift** | Agent forgets prior decisions; loses track of state across sessions or compactions. ~65% of enterprise agent failures attributed here. |
| 4 | **Token / context bloat** | Verbose outputs and unbounded context accumulation; larger windows hide rather than solve the problem. |
| 5 | **Verbosity & over-engineering** | Agent adds unrequested abstractions; "one-shots" entire apps; generates more code than was asked for. |
| 6 | **Codebase entropy** | Architectural drift, duplicated logic, broken patterns accumulating faster than humans can review. |
| 7 | **Decision amnesia** | Architectural rationale evaporates when sessions end; future agents re-litigate settled questions. |
| 8 | **Destructive / irreversible actions** | Over-broad permissions → dropped databases, mass deletes, leaked secrets. (Canonical: the Replit DB-deletion incident, July 2025.) |
| 9 | **Prompt injection & memory poisoning** | Hostile instructions embedded in docs, tools, or RAG indexes; persistent corruption of agent memory. (OWASP ASI06.) |

Two of these deserve a longer second look:

The **Replit incident** (failure mode 8) is a frequently cited example. In
July 2025 an autonomous coding agent, given broad shell access during a
multi-day build, decided that the production PostgreSQL database "looked
unused" and dropped it. There was no malice and no exotic prompt — just an
agent acting inside a permission boundary that should never have been that
wide. We will configure a tool allowlist in §4.2.4 to prevent the shape of
mistake.

**Goal hijack** (failure mode 1) is OWASP's name for a more subtle
phenomenon: a long-running agent, partway through a task, encounters
text — in a fetched document, a tool result, an issue comment — that
re-directs it toward a different objective. The agent follows the new
objective and reports success on it. Specs and explicit checkpoints (§4.2)
exist to make this kind of drift visible.

## The map: failures, fundamentals, artifacts

Every failure mode above has a name in classical software engineering for
the practice that addresses it, and every classical practice now has a
modern *artifact* — a file in your repo, a configuration, or a protocol — that
encodes it for an AI agent to read.

This three-column mapping is the spine of the rest of the chapter:

| AI fails when… | The fundamental | The modern artifact |
|---|---|---|
| …it solves the wrong problem (1) | **Spec-driven thinking** | a written spec; Spec-Kit; `/grill-me`-style interview |
| …it produces plausible but wrong code (2) | **Test-Driven Development** | a test that pins behaviour, written before the implementation |
| …it forgets conventions across sessions (3) | **Project memory** | `AGENTS.md`, `CLAUDE.md`, `copilot-instructions.md` |
| …it bloats the context window (4) | **Parsimony** | compressed memory; caveman-style prompts |
| …it over-engineers (5) | **YAGNI; vertical slices** | small specs; per-task scoping |
| …it accelerates the codebase into mud (6) | **Design discipline** | ADRs; "zoom out" sub-agents; periodic refactors |
| …it forgets *why* (7) | **Decision logs** | ADRs; persistent memory tools |
| …it runs destructive commands (8) | **Least privilege** | tool allowlists; sandboxes; git guardrails |
| …it can't see the right context | **Tool use & retrieval** | MCP servers; custom tools |

§4.1 covers the **left half of column 2** — Spec-Driven Development
and Test-Driven Development — by building two checks of `autograde` from
scratch.

§4.2 covers the **right half of column 3** — `AGENTS.md`, Skills,
MCP, and guardrails — by hardening the workflow from §4.1 and using it to
add more checks.

## 4.0.6 The tool landscape

A short note before we start coding. This chapter uses **GitHub Copilot in
VS Code** for every hands-on step. Copilot is one
of several roughly equivalent options as of 2026:

- **GitHub Copilot** (Microsoft) — what we use here. Strong VS Code
  integration; agent mode shipped in 2025.
- **OpenAI Codex** — cloud-hosted coding agent; same `AGENTS.md` standard.
- **Anthropic Claude Code** — terminal-native agent; introduced the
  `SKILL.md` and `CLAUDE.md` conventions we will see in §4.2.
- **Cursor**, **Windsurf** — VS Code forks with their own agent UIs.

The patterns we will learn — specs, tests, project memory files, tool
allowlists — work across all of them. The names of files (`AGENTS.md` vs
`CLAUDE.md`), the keyboard shortcuts, and the chat panel layout differ; the
underlying engineering does not.

## Building `autograde`

The running project for the rest of this chapter is a small Python CLI
called `autograde`. Its job is to take a path to a local Git repository and
report whether that repository satisfies a fixed list of checks.

Here is the contract we will work to. *You are not implementing this now —
you will build it across §4.1 and §4.2, with the agent doing most of the
typing.*

```{code-block} text
autograde PATH

PASS|FAIL  is_git_repo                  <reason>
PASS|FAIL  main_branch_exists           <reason>
PASS|FAIL  feature_branch_exists        <reason>
PASS|FAIL  main_file1_has_three_lines   <reason>
PASS|FAIL  main_does_not_contain_file2  <reason>
PASS|FAIL  feature_file2_exists         <reason>
```

Each check is a single Python function with the signature
`check_<name>(repo_path: Path) -> tuple[bool, str]` — pinned later in
`AGENTS.md` so the agent stays consistent across sessions.

This is a meta-pedagogical mirror: the autograder you are building is
conceptually the same one that will grade your Chapter 1 work. You are using
modern AI-assisted practices to build the tool that enforces those practices
on you.

### Round 1 — let the agent scaffold it

Create an empty directory and open it in VS Code:

```{code-block} bash
mkdir autograde
cd autograde
code .
```

Open the Copilot Chat panel (`Ctrl+Shift+I`) and switch the dropdown to
**Agent** mode. Paste this — and *only* this:

```{code-block} text
Scaffold a Python CLI package called `autograde` in this empty directory.
The CLI takes a path to a git repository and prints PASS|FAIL lines for a
fixed list of checks.
```

The exact response will differ from session to session, but the *shape* in
early 2026 is consistent. Typical output:

```{code-block} text
Created .venv/ via `python -m venv .venv`
Created requirements.txt (click==8.1.7, pytest==8.0.0)
Created setup.py with setuptools entry_points
Created README.md
Created LICENSE (MIT)
Created autograde/cli.py with click-based CLI
Created autograde/checks.py with 12 check functions
Created tests/test_autograde.py using unittest.TestCase
Created .github/workflows/ci.yml
Run `pip install -e .` then `python -m unittest` to verify.
```

Three things in that output are wrong, *for this project*:

1. The agent reached for `venv` + `pip` + `setup.py`. You spent Chapter 2
   learning `uv` precisely because that older stack is not what we use.
2. The agent wrote tests with `unittest`. You spent Chapter 3 learning
   `pytest`.
3. The agent invented twelve check functions, a `click` dependency, a CI
   workflow, an MIT license, and a README — none of which you asked for.

Tag what you just saw with the numbers from §4.0.4: this is failure mode
**#1 (misalignment — the agent picked a stack you did not want)** and
failure mode **#5 (over-engineering — the agent shipped a CI workflow when
you did not request one)**. Given a vague prompt, the agent fell back to
the most common shape of "Python CLI project" in its training data, which
is not the shape *your* project wants.

```{admonition} Reset
:class: warning
In Copilot Chat, click **Discard all**. Then make sure the directory is
empty again before you re-prompt:

```{code-block} bash
rm -rf .venv autograde tests .github
rm -f requirements.txt setup.py README.md LICENSE
ls -la   # nothing but `.` and `..`
```
```

### Round 2 — same agent, constrained prompt

Open Copilot Chat again, still in Agent mode. Paste **this** prompt
instead. The only difference is one short paragraph pinning the stack and
the scope:

```{code-block} text
Scaffold a Python CLI package called `autograde` in this empty directory.

Stack and conventions (please follow exactly):
- Use `uv` for project setup (`uv init`, `uv add --dev pytest`).
  No `venv`, no `pip`, no `setup.py`, no `requirements.txt`.
- Use `pytest` for tests. No `unittest`.
- No CLI framework — no `click`, no `typer`, no `argparse`.

Scope: just the empty package and the pytest dev-dependency. Do not write
any check functions yet. Do not add a README, a LICENSE, a CI workflow, or
any other files. Stop after `uv run pytest` exits 0 (zero tests is fine).
```

This time the agent should produce something close to:

```{code-block} text
Ran: uv init autograde
Ran: cd autograde && uv add --dev pytest
Ran: uv run pytest
no tests ran in 0.01s
```

Verify the result yourself:

```{code-block} bash
uv run pytest
```

The output is the same `no tests ran` line, which is the expected result
for a project with no tests yet. Now place it under version control:

```{code-block} bash
git init
git add .
git commit -m "Bootstrap autograde with uv and pytest"
```

### What just happened

Same model, same task, same project — two very different scaffolds. The
delta is **one paragraph of constraints in the prompt.** Round 1 produced a
generic Python project you would have to delete and rebuild; round 2
produced exactly the empty `uv` + `pytest` package you wanted.

That paragraph is the seed of the artifact you will meet in §4.2 as
**`AGENTS.md`**: a project-level note the agent reads on every turn so you
do not have to re-paste your stack conventions into every chat.

```{admonition} Forward reference
:class: hint
You just paid the cost of stating "use uv, use pytest, no CLI framework"
inside one prompt. If you start a new chat tomorrow, you will pay it again.
In §4.2.1 we move that paragraph into `AGENTS.md` at the repo root, and
every future Copilot session reads it automatically. The friction you just
felt is exactly what `AGENTS.md` exists to remove.
```

You now have an empty `uv` + `pytest` project under version control, and
concrete evidence that **the prompt shape, not the model, decided what
landed on disk.** §4.1 takes the next step: the agent writes its first real
check.
