## 4.1 Software Development Practices and Specifications

### Spec-Driven Development

Since the 1960s, software development has been plagued by the "spec problem": the difficulty of writing and maintaining specifications that accurately capture the intended behaviour of a system. The spec problem is not just a documentation problem; it is a communication problem between stakeholders, developers, and the system itself. Specifications are often incomplete, outdated, or ignored, leading to misalignment between what the system does and what it is supposed to do.

The phrase "specification-driven development" first appears in the published record at XP 2004 (Ostroff, Paige, Makalsky), but the idea has been around in various forms for much longer. The core insight is that specifications should drive development, not the other way around. In a spec-driven approach, you write a specification that describes the desired behaviour of the system before you write any code. The specification serves as a contract that guides development and provides a basis for testing and verification.

The AI-Driven Renaissance (2024-Present)
AI coding assistants created the conditions for spec-driven development to finally become mainstream. Not because AI solved the spec problem, but because AI made the spec problem impossible to ignore. With AI coding, code generation becomes cheap. Errors compound across generations of AI output, context gets lost between sessions, there's no persistent model of intent, and the resulting code is untraceable. You can't explain why it works, and you can't explain why it breaks.

In response, multiple tools launched in rapid succession, all converging on the same insight: you need specifications before you need code.

These new tools are a huge step forward, but they don't solve the problem of the "executability gap": a spec that is not executable is a spec that cannot be verified. You can write a markdown spec, but if you can't run it, how do you know it's right? The next section will show how to use Test-Driven Development to create executable specifications that keep your AI-generated code aligned with your intent.

### Test-Driven Development

Kent Beck popularized Test-Driven Development around 2000. TDD reframed testing as a design activity, not just verification. Your tests described what the code should do before the code existed. That made tests a form of specification. 

TDD is a natural fit for AI coding because it provides a way to create executable specifications that can be verified automatically. With TDD, you write a test that describes the desired behaviour of the system, and then you write the code to make that test pass. The test serves as a contract that guides development and provides a basis for verification. TDD also provides a safety net for refactoring, allowing you to improve the design of your code without fear of breaking existing functionality.

The combination of spec-driven development and TDD creates a powerful feedback loop that keeps your AI-generated code aligned with your intent. The spec provides the high-level guidance, while the tests provide the low-level checks that ensure the code does what it's supposed to do. 

TDD is based on a simple cycle that repeats for each new piece of functionality you want to add to your system and it is know as the "Red-Green-Refactor" cycle:

- **Red.** Write a small test for the next slice of behaviour. Run it. It
  fails — usually because the function it calls does not yet exist. The
  test bar goes red.
- **Green.** Write the smallest amount of code that makes that test pass.
  Nothing more. Run the suite again; the bar goes green.
- **Refactor.** With a green bar as a safety net, improve the structure of
  the code (rename, extract, deduplicate) without changing what it does.
  Re-run the suite after each change to confirm the bar stays green.

The discipline feels backwards at first, but it buys two things: each new piece of production code starts life with a test that pins its behaviour, and you always have a fast answer to *"did the last change break anything?"*

But how we incorporate TDD into our work with AI coding assistants? The naive approach is to just tell the agent "use TDD". In practice, that is a recipe for failure. The agent will often write tests that are too broad, too narrow, or not actually testing the behaviour you care about. The result is a false sense of security: you have tests, but they don't actually verify the correctness of your code.

A March 2026 paper, *TDD Anti-Patterns in Agentic Development* (arXiv 2603.17973), showed that a vague instruction to "use TDD" can be worse than no TDD instruction at all. Agents tend to add excessive mocking, pin the implementation instead of the behaviour, and refactor themselves into brittle designs.
