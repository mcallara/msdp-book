# 4.2 Dependency Management and Virtual Environments

## Dependency Management

In a `uv` project, dependency management and virtual environment management are part of the same workflow. You declare the dependencies in `pyproject.toml`, `uv` records the exact resolved versions in `uv.lock`, and installs them into the project's `.venv` directory.

For a beginner, the most useful mental model is this:

- `pyproject.toml` describes what your project needs;
- `uv.lock` records the exact versions that were resolved;
- `.venv` contains the environment where those dependencies are installed.

### The Default Workflow

Suppose we want to add a small script to our package that uses `pandas`. First, create a new file called `analysis.py` in `src/my_package` with the following content:

```{code-block} python
import pandas as pd
df = pd.DataFrame({'A': [1, 2, 3], 'B': [4, 5, 6]})
```

```{code-cell} bash
:tags: ["remove-input","remove-output"]
cd src
cd my_package
echo $'import pandas as pd
df = pd.DataFrame({"A": [1, 2, 3], "B": [4, 5, 6]})' > analysis.py
cd ..
cd ..
```

Now add `pandas` as a dependency:

```{code-cell} bash
:tags: ["remove-input","remove-output"]
uv add pandas
```

```{code-block} bash
uv add pandas
```

```{admonition} What to notice
:class: hint 
- `uv add` updates `pyproject.toml`.
- `uv add` also updates the lockfile and the project environment.
- You do not need to run a separate install command after `uv add`; the project is kept in sync automatically.
```

This is the core `uv` workflow for dependencies: declare the requirement once, and let `uv` update the rest of the project state for you.

### What Changes in the Project Files

After adding `pandas`, `uv` writes the dependency to the `project.dependencies` field in `pyproject.toml`. In a package project, the file looks broadly like this:

```{code-block} toml
[project]
name = "my-package"
version = "0.1.0"
description = ""
readme = "README.md"
requires-python = ">=3.12"
dependencies = [
    "pandas>=2.3.1,<3"
]

[build-system]
requires = ["uv_build>=0.6,<0.7"]
build-backend = "uv_build"
```

```{admonition} What to notice
:class: hint 
- Runtime dependencies belong in `project.dependencies`.
- `uv` chooses a compatible version constraint automatically when you use `uv add`.
- The exact resolved versions are stored in `uv.lock`, which should be committed to version control.
```

`uv.lock` should not be edited manually. It is generated and maintained by `uv` so that installations remain reproducible across machines.

### Running Code in the Project Environment

The recommended way to run commands in a `uv` project is with `uv run`. For example:

```{code-block} bash
uv run python src/my_package/analysis.py
```

Before running the command, `uv` checks that the lockfile matches `pyproject.toml` and that the environment matches the lockfile. This means `uv run` acts as the default "just run my project" command.

### Development Dependencies and Dependency Groups

Not every dependency belongs in the published requirements of the package. Tools such as `pytest`, `ruff`, or documentation builders are typically local development dependencies.

To add `pytest` to the default development group, run:

```{code-cell} bash
:tags: ["remove-input","remove-output"]
uv add --dev pytest
```

```{code-block} bash
uv add --dev pytest
```

This creates a `dev` dependency group in `pyproject.toml`:

```{code-block} toml
[dependency-groups]
dev = [
    "pytest>=8.4.1,<9"
]
```

Development dependencies are local to the project and are not published as part of the package requirements.

If you want a different group, such as `lint`, you can create it directly:

```{code-block} bash
uv add --group lint ruff
```

```{admonition} What to notice
:class: hint
- `project.dependencies` contains the dependencies required to run the project.
- `dependency-groups` contains local development dependencies.
- In `uv`, the `dev` group is included by default when running `uv sync` or `uv run`.
```

### Optional Dependencies

If your package offers an extra feature that users may or may not want, use optional dependencies instead of a development group. For example:

```{code-block} bash
uv add --optional plotting matplotlib
```

This stores the dependency in `project.optional-dependencies`. Optional dependencies are part of the package metadata, while development groups are local-only.

### Virtual Environments in uv

In `uv`, the project environment lives in a `.venv` directory inside the project. Unlike the Poetry workflow, you usually do not need to configure where the environment should go: `uv` uses the project environment by default.

The `.venv` directory and `uv.lock` are created the first time you run a project command such as:

```{code-block} bash
uv run python --version
```

or:

```{code-block} bash
uv sync
```

### Using `uv run` vs `uv sync`

There are two common ways to work with the project environment.

The first is the simplest:

```{code-block} bash
uv run python src/my_package/analysis.py
```

This is the recommended default for beginners because `uv` handles synchronization automatically before running the command.

The second is more explicit:

```{code-block} bash
uv sync
source .venv/bin/activate
python src/my_package/analysis.py
```

Use this pattern when you want to activate the environment and run several commands in the same shell session.

### Syncing Specific Dependency Groups

Once groups are defined, you can choose which groups to include when syncing the environment.

```{code-block} bash
# Sync only the project's runtime dependencies
uv sync --no-default-groups

# Sync only the dev group
uv sync --only-group dev

# Sync the default groups plus an additional group
uv sync --group lint

# Exclude a specific group
uv sync --no-group dev
```

This is useful when you want a smaller environment for CI, or when different tasks require different tool sets.

### Updating and Removing Dependencies

To remove a dependency:

```{code-block} bash
uv remove pandas
```

To upgrade one package while keeping the rest of the lockfile stable:

```{code-block} bash
uv lock --upgrade-package pandas
```

To import dependencies from an existing requirements file during migration:

```{code-block} bash
uv add -r requirements.txt
```

### Understanding Version Constraints

`uv` stores dependency constraints using standard Python dependency specifiers. For example:

```{code-block} toml
[project]
dependencies = [
    "pandas>=2.3.1,<3"
]

[dependency-groups]
dev = [
    "pytest>=8.4.1,<9"
]
```

Here, `pandas>=2.3.1,<3` means any compatible version in the `2.x` series. `pytest>=8.4.1,<9` means any compatible `8.x` release. These ranges give some flexibility while preventing accidental upgrades to incompatible major versions.

## Version Control Best Practices

Always commit `uv.lock` to version control. This ensures that everyone working on the project resolves and installs the same dependency versions.
