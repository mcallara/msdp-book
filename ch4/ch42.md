# 4.2 Dependency Management and Virtual Environments

## Dependency Management

In a `uv` project, dependency management and virtual environment management are part of the same workflow. You declare the dependencies in `pyproject.toml`, `uv` records the exact resolved versions in `uv.lock`, and installs them into the project's `.venv` directory.

`uv` keeps these files in sync automatically. When you add a dependency with `uv add`, it updates `pyproject.toml`, resolves the new dependency graph, updates `uv.lock`, and installs the new dependencies into `.venv`. You do not need to run separate commands to update the lockfile or install dependencies.

In summary, the key files for dependency management in a `uv` project are:
- `pyproject.toml` describes what your project needs;
- `uv.lock` records the exact versions that were resolved;
- `.venv` contains the environment where those dependencies are installed.

### Adding a Dependency to the Project

Suppose we want to add a small script to our package that uses `pandas`. First, create a new file called `analysis.py` in `src/my_package` with the following content:

```{code-block} python
import pandas as pd
df = pd.DataFrame({'A': [1, 2, 3], 'B': [4, 5, 6]})
```

```{code-cell} bash
:tags: ["remove-input","remove-output"]
cd src
cd my_package
echo $'import pandas as pd
df = pd.DataFrame({"A": [1, 2, 3], "B": [4, 5, 6]})' > analysis.py
cd ..
cd ..
```

Now add `pandas` as a dependency:

```{code-cell} bash
:tags: ["remove-input","remove-output"]
uv add pandas
```

```{code-block} bash
uv add pandas
```

```{admonition} What to notice
:class: hint 
- `uv add` updates `pyproject.toml`.
- `uv add` also updates the lockfile and the project environment.
- You do not need to run a separate install command after `uv add`; the project is kept in sync automatically.
```

After adding `pandas`, `uv` writes the dependency to the `project.dependencies` field in `pyproject.toml`. In a package project, the file looks broadly like this:

```{code-block} toml
[project]
name = "my-package"
version = "0.1.0"
description = ""
readme = "README.md"
requires-python = ">=3.12"
dependencies = [
    "pandas>=2.3.1,<3"
]

[build-system]
requires = ["uv_build>=0.6,<0.7"]
build-backend = "uv_build"
```

```{admonition} What to notice
:class: hint 
- Runtime dependencies belong in `project.dependencies`.
- `uv` chooses a compatible version constraint automatically when you use `uv add`.
- The exact resolved versions are stored in `uv.lock`, which should be committed to version control.
```

`uv.lock` should not be edited manually. It is generated and maintained by `uv` so that installations remain reproducible across machines.

### Understanding Version Constraints

`uv` stores dependency constraints using standard Python dependency specifiers. For example:

```{code-block} toml
[project]
dependencies = [
    "pandas>=2.3.1,<3"
]

[dependency-groups]
dev = [
    "pytest>=8.4.1,<9"
]
```

The syntax `>=2.3.1,<3` means "at least version 2.3.1 but before 3.0." Similarly, `pytest>=8.4.1,<9` means "at least 8.4.1 but before 9.0." These ranges allow for patch and minor updates (which typically contain bug fixes) while preventing jumps to major versions. We'll explore the principles behind versioning in a later chapter.

## Version Control Best Practices

Always commit `uv.lock` to version control. This ensures that everyone working on the project resolves and installs the same dependency versions.

Do not commit `.venv`. The virtual environment is machine-local state that `uv` can recreate from `pyproject.toml` and `uv.lock`, so it should stay out of version control.

### Updating and Removing Dependencies

To remove a dependency:

```{code-block} bash
uv remove pandas
```

To upgrade one package while keeping the rest of the lockfile stable:

```{code-block} bash
uv lock --upgrade-package pandas
```

To import dependencies from an existing requirements file during migration:

```{code-block} bash
uv add -r requirements.txt
```

### Running Code in the Project Environment

The recommended way to run commands in a `uv` project is with `uv run`. For example:

```{code-block} bash
uv run python -m my_package.analysis
```

Before running the command, `uv` checks that the lockfile matches `pyproject.toml` and that the environment matches the lockfile. This means `uv run` acts as the default "just run my project" command.

### Organizing Dependencies with Dependency Groups

As projects grow, you'll work with many dependencies serving different purposes. Your package depends on `pandas` for runtime functionality, but you also need `pytest` for testing and `sphinx` for documentation. These development tools are essential *during development* but shouldn't be forced onto users who install your package. Organizing dependencies into groups lets you keep these concerns separate.

Not every dependency belongs in the published requirements of the package. Tools such as `pytest` or documentation builders are typically local development dependencies.

To add `pytest` to the default development group, run:

```{code-cell} bash
:tags: ["remove-input","remove-output"]
uv add --dev pytest
```

```{code-block} bash
uv add --dev pytest
```

This creates a `dev` dependency group in `pyproject.toml`:

```{code-block} toml
[dependency-groups]
dev = [
    "pytest>=8.4.1,<9"
]
```

Development dependencies are local to the project and are not published as part of the package requirements.

### Optional Dependencies

If your package offers an extra feature that users may or may not want, use optional dependencies instead of a development group. For example:

```{code-block} bash
uv add --optional plotting matplotlib
```

This stores the dependency in `project.optional-dependencies`. Optional dependencies are part of the package metadata, while development groups are local-only.

### Custom Dependency Groups

You can also create custom groups for different purposes. For example, you might have a `lint` group for linters and formatters:

```{code-block} bash
uv add --group lint black flake8
```

## Virtual Environments in uv

In `uv`, the project environment lives in a `.venv` directory inside the project. This is a common convention that keeps the environment self-contained within the project directory. The `.venv` directory is created automatically when you run a project command for the first time, such as `uv run` or `uv sync`. You do not need to create it manually. The `.venv` directory is where `uv` installs the dependencies specified in `pyproject.toml` and resolved in `uv.lock`.

The `.venv` directory and `uv.lock` are created the first time you run a project command such as:

```{code-block} bash
uv run python --version
```

or:

```{code-block} bash
uv sync
```

### Running Commands in the Project Virtual Environment

When you run a command with `uv run`, it automatically uses the Python interpreter from the `.venv` environment. This means that any dependencies installed in the project environment will be available when you run commands with `uv run`. For example, after adding `pandas` as a dependency, you can run the `analysis.py` script with:

```{code-block} bash
uv run python -m my_package.analysis
```

`uv` ensures that the environment is synchronized before running the command. This means that `uv` installs any missing dependencies before running the command.

If you need to work in the environment interactively, you can manually sync the environment and activate it with:

```{code-block} bash
uv sync
source .venv/bin/activate
python -m my_package.analysis
```

You can use this pattern when you want to activate the environment and run several commands in the same shell session.

### Syncing Specific Dependency Groups

By default, `uv sync` installs the project's main runtime dependencies plus any groups marked as default (like `dev`). Sometimes you need finer control—for example, in a CI pipeline that only needs to test the package without development tools.

Once groups are defined, you can choose which groups to include when syncing the environment.

```{code-block} bash
# Sync only the project's runtime dependencies (exclude all groups)
uv sync --no-default-groups

# Sync only the dev group (useful for CI that needs testing tools)
uv sync --only-group dev

# Sync runtime dependencies plus a custom group
uv sync --group lint

# Sync everything except a specific group
uv sync --no-group dev
```

This is useful when you want a smaller environment for CI, or when different tasks require different tool sets.

### Remember

- **pyproject.toml**: Declares what your project needs
- **uv.lock**: Records the exact versions that work together
- **.venv**: The isolated environment where dependencies are installed
- Commit `.lock` to git; add `.venv` to `.gitignore`
