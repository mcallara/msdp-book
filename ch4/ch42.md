# 4.2 Artifacts, Tools, and Guardrails

In the previous section we focused on practices: write specs before code, and
use tests to make behavior executable. Those practices work best when they are
supported by artifacts that survive beyond a single chat session.

This section is about those artifacts. In an AI-assisted workflow, the editor is
not just a text editor. It is a workspace where the model reads files, follows
instructions, calls tools, and sometimes performs actions on your behalf. That
means part of the software design now lives in repository metadata and editor
configuration.

For Chapter 4, four artifacts matter most:

- repository instructions in `AGENTS.md`,
- reusable prompts and skills,
- tool connections such as MCP servers,
- and approval settings that enforce least privilege.

## `AGENTS.md`: persistent repository memory

One of the recurring failure modes in AI coding is decision amnesia. You tell an
agent about naming conventions, testing expectations, or workflow constraints in
chat, and then the model forgets them in the next session. The result is that
the same guidance has to be repeated over and over.

An `AGENTS.md` file addresses that problem by storing stable repository-level
instructions in a place the agent can consult as part of its working context.
You can think of it as a lightweight operating manual for collaborators that are
part human and part machine.

A useful `AGENTS.md` file typically includes:

- the project purpose,
- key architectural conventions,
- how to run tests and validation commands,
- scope boundaries,
- and review rules such as "do not modify generated files" or "prefer small patches."

For the `autograde` project, an initial version might contain statements such as:

```md
# AGENTS.md

- Use `pytest` for verification.
- Add one repository check at a time.
- Prefer behavioral tests over implementation-detail tests.
- Do not rewrite existing specs; add a new numbered spec instead.
- Ask before changing GitHub workflow files.
```

The file does not need to be long. Its value comes from precision and stability,
not volume.

## Reusable prompts and skills

Another common source of waste in AI-assisted development is repeated prompting.
If you find yourself giving the same careful instruction multiple times, that is
usually a sign that the instruction should become a reusable artifact.

In VS Code, this can take several forms:

- a saved prompt file for recurring review or implementation tasks,
- a skill that packages a reusable workflow,
- or project instructions that standardize how the agent should behave.

Suppose you repeatedly want the agent to challenge the quality of a generated
patch before you accept it. Instead of typing the same message every time, you
can store a prompt such as `grill-check.prompt.md` that says:

```md
Review the proposed change critically.
List behavioral regressions, missing tests, unsafe assumptions, and scope creep.
Do not summarize until after the findings.
```

Once captured as an artifact, that workflow stops depending on memory and starts
depending on the repository.

```{admonition} Artifact or chat message?
:class: tip
If an instruction matters only once, keep it in chat. If it matters every time,
promote it into a file.
```

## Tools and MCP servers

Modern coding agents are more useful when they can do more than predict text.
They may search the workspace, run tests, read web pages, inspect terminals, or
call external services. In VS Code, these capabilities are exposed through tools.

Broadly, you can think of three categories:

- **built-in tools** provided by the editor,
- **extension tools** exposed by installed extensions,
- **MCP tools** provided by Model Context Protocol servers.

MCP matters because it gives the agent structured access to capabilities outside
the local editor. For example, a fetch server can retrieve authoritative online
documentation during a task instead of relying on the model's memory of that
documentation.

That matters for reliability. A model may vaguely remember a command-line flag
or API behavior. A fetch tool lets it check.

In this chapter's lab, a fetch MCP server is used to retrieve documentation from
sites such as `git-scm.com` during the `autograde` workflow. The important point
is not the specific server. The important point is the pattern: when correctness
depends on external facts, prefer tool-assisted retrieval over model memory.

## Approvals and least privilege

Tools make agents more capable, but also more dangerous. An agent with shell
access, network access, and file-system write access can do useful work quickly.
It can also make damaging decisions quickly.

For that reason, approval settings are not a nuisance. They are part of the
design of the system.

The principle to follow is the same one used elsewhere in security engineering:
**least privilege**. Give the agent the smallest set of permissions that still
allows useful progress.

That usually means:

- read-only access by default,
- explicit approval for destructive or irreversible actions,
- narrow tool scopes where possible,
- and human review before changes that affect production systems, secrets, or history.

This is the engineering response to failure mode 8 from [ch4/ch40.md](/var/home/core/dev/msdp-book/ch4/ch40.md): destructive actions are less likely when dangerous operations are difficult to trigger accidentally.

## A practical checklist for the `autograde` project

By the end of this chapter, the repository should contain a small but effective
control surface for working with the agent:

- `specs/` with numbered specs for each implemented check,
- `tests/` with executable verification,
- `AGENTS.md` with project conventions,
- at least one reusable prompt or skill for repetitive tasks,
- and tool settings that let the assistant inspect what it needs without granting unnecessary power.

Notice how these artifacts work together:

- Specs reduce ambiguity.
- Tests catch regressions.
- `AGENTS.md` preserves local conventions.
- Reusable prompts standardize recurring workflows.
- Tools retrieve facts and execute checks.
- Approvals keep automation bounded.

This is what controlled AI-assisted development looks like in practice. The goal
is not to remove the human from the loop. The goal is to make the loop explicit,
repeatable, and cheap to audit.
